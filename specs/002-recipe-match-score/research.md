## Finding 1 — Vector Representation: Local TF-IDF

**Question**: How should recipe text be converted into a vector for similarity comparison, without an external embeddings API?

**Decision**: Local TF-IDF (Term Frequency–Inverse Document Frequency) computed entirely on-device in `src/utils/tfidf.js`.

**Algorithm**:

```
tfidf(text, corpus) → number[]

For each term t in vocabulary(corpus):
  TF(t, text)  = count(t in text) / total_terms(text)
  IDF(t)       = log((|corpus| + 1) / (df(t, corpus) + 1)) + 1   [smooth IDF]
  tfidf(t)     = TF(t, text) × IDF(t)

vocabulary(corpus) = sorted unique tokens across all documents in corpus
Result: dense vector indexed by vocabulary(corpus), length = |vocabulary(corpus)|
```

**Tokenization**: lowercase, split on `/[^a-z]+/`, discard tokens of length < 2.

**IDF formula**: Smooth IDF — adds 1 to numerator and denominator to prevent division-by-zero for unseen terms, and adds 1 to the log result to keep all IDF values ≥ 1.

**Rationale**:
- No external API calls, no secrets, fully offline — simpler than any embedding service.
- TF-IDF captures lexical overlap in recipe vocabulary (ingredient names, cuisine tags, cooking characteristics), which is the dominant signal for recipe similarity in this use case.
- Pure functions are trivially testable.
- No new npm packages — standard JS math only.

**Limitations accepted**:
- Does not capture semantic synonyms (e.g., "spicy" vs "hot"). This is acceptable: recipe tags are controlled vocabulary, and recipe names/descriptions are already consistent in style (generated by Claude).
- Vector dimension grows with corpus size. Manageable at typical usage (hundreds of recipes) — see Finding 6.

**Alternatives considered**:
- Voyage AI REST API (Anthropic's recommended embeddings partner) — rejected; requires a new API key (`VOYAGE_API_KEY`), network dependency, and adds latency to batch scoring. The feature should be fully offline.
- Pre-trained word embeddings bundled in the app (e.g., GloVe 50d) — rejected; adds significant bundle size for marginal quality gain on short recipe texts with controlled vocabulary.

## Finding 2 — Cosine Similarity Algorithm

**Decision**: Pure JavaScript implementation, unchanged from original design. No library.

**Formula**:

```
cosineSimilarity(a, b) = dot(a, b) / (norm(a) × norm(b))
```

**Edge cases**: zero-norm vector → return `0`. Length mismatch → throw `Error('Vector length mismatch')`.

**Percentage conversion**: `Math.max(0, Math.round(similarity * 100))` — clamp negative values to 0%.

## Finding 3 — Taste Profile: Running Mean with Vocabulary

**Question**: How should the taste profile vector be updated on right-swipe, given that the TF-IDF vocabulary grows as more recipes are seen?

**Decision**: Store `{ vector: number[], vocabulary: string[], count: number }`. The vocabulary is stored alongside the vector so new vectors can always be aligned to the same index space before computing the running mean.

**Update algorithm** (in `updateTfidfProfile(likedText, currentCorpus)`):

```
1. newVocabulary = buildVocabulary(currentCorpus)
2. newVector     = tfidf(likedText, currentCorpus)   // indexed by newVocabulary

If profile exists:
3. Align profile.vector to newVocabulary:
   alignedOld[i] = profile.vector[profile.vocabulary.indexOf(term)] if term in profile.vocabulary, else 0
4. newMean[i] = (alignedOld[i] × count + newVector[i]) / (count + 1)
5. Store { vector: newMean, vocabulary: newVocabulary, count: count + 1 }

If no profile:
3. Store { vector: newVector, vocabulary: newVocabulary, count: 1 }
```

**Alignment semantics**: When the vocabulary grows (new terms appear in the corpus), the stored profile vector is zero-padded at positions for those new terms. This is conservative — no preference is attributed to terms not yet seen in liked recipes, which is correct.

**Running mean formula**: `new_mean[i] = (old_mean[i] × count + new_value[i]) / (count + 1)` (element-wise).

**Storage shape**: `{ vector: number[], vocabulary: string[], count: number }`
**AsyncStorage key**: `@srp/tfidf_profile`
**Null state**: key absent = no likes yet.

## Finding 4 — Corpus Management

**Question**: How is the corpus (all seen recipe texts) maintained and stored, and when should it be updated?

**Decision**: Store the corpus as `string[]` at `@srp/recipe_corpus`. Update it when each batch loads (5 new texts added), before scores are computed.

**Rationale**: Correct IDF requires the full seen-recipe corpus, not just liked recipes. Updating at batch-load time (rather than only on right-swipe) ensures IDF values reflect all the recipes the user has encountered — making scores more meaningful as the corpus grows.

**Read pattern**: `getRecipeCorpus()` is called at batch-load time and at right-swipe time. Both reads are `AsyncStorage.getItem` calls — fast and non-blocking.

**Storage**: Stored in `tfidfProfile.js` alongside the profile, using a separate key (`@srp/recipe_corpus`). Both are managed by the same module to keep related storage logic co-located.

**Corpus size estimate**:
- Average recipe text length: ~150 characters
- 50 batches of 5 = 250 recipes total → ~37 KB
- 200 batches (heavy user) = 1000 recipes → ~150 KB
- Both well within AsyncStorage limits for mobile apps.

**Deduplication**: Recipe texts are not deduplicated in the corpus (same text could appear twice if a recipe is regenerated). This is acceptable — slight IDF distortion from duplicates is negligible.

## Finding 5 — Batch Scoring: Synchronous, No Parallelism Needed

**Question**: How should match scores for 5 recipes be computed efficiently?

**Decision**: Synchronous loop — no `Promise.allSettled` or async needed.

**Rationale**: TF-IDF and cosine similarity are pure synchronous computations. The corpus and profile are loaded once from AsyncStorage (one async call each), then all 5 recipe vectors are computed in a simple loop. No parallelism is needed or helpful for CPU-bound synchronous work.

**Revised `loadBatch` sequence**:

```
1. fetchRecipes()        — Claude API (existing async)
2. fetchBatchImages()    — Unsplash API (existing async, Promise.allSettled)
3. getRecipeCorpus()     — AsyncStorage read (async)
4. addToRecipeCorpus(5 new texts) — AsyncStorage write (async)
5. getTfidfProfile()     — AsyncStorage read (async)
6. If profile exists:
     for each recipe:
       vector = tfidf(recipeText, corpus)   — synchronous
       projected = project(vector, profile.vocabulary)  — synchronous
       score = cosineSimilarity(projected, profile.vector)  — synchronous
       matchScore = Math.max(0, Math.round(score * 100))
7. Build cards: { recipe, imageUrl, matchScore }
```

**On right-swipe**:

```
1. updateTagProfile()    — existing
2. getRecipeCorpus()     — AsyncStorage read
3. updateTfidfProfile(likedText, corpus) — AsyncStorage read+write (handles vector+vocabulary update)
   — failure is silent (logged, not surfaced to user)
```

**Error handling**: If any step in the scoring path throws, `matchScore` is set to `null` for all cards. This is caught with a single `try/catch` around steps 3–6 in `loadBatch`.

## Finding 6 — Vocabulary Projection for Scoring

**Question**: When scoring new recipes at batch-load time, the live corpus vocabulary may be larger than the stored profile's vocabulary. How are these aligned for cosine similarity?

**Decision**: Project the new recipe's TF-IDF vector down to the profile's stored vocabulary before computing similarity.

**Algorithm**:

```
score(newRecipeText, profile, corpus):
  newVector = tfidf(newRecipeText, corpus)       // indexed by buildVocabulary(corpus)
  liveVocab = buildVocabulary(corpus)
  projected = profile.vocabulary.map(term =>
    liveVocab.includes(term) ? newVector[liveVocab.indexOf(term)] : 0
  )
  return cosineSimilarity(projected, profile.vector)
```

**Why this works**: Cosine similarity is computed in the profile's vocabulary subspace. Terms in the new recipe that aren't in the profile's vocabulary are ignored (projected to 0) — meaning they don't contribute to the score. This is semantically correct: if the profile has never encountered a term, it has no preference for it.

**Indexing efficiency**: At small vocabulary sizes (hundreds of terms), linear `indexOf` is fine. If vocabulary grows to thousands of terms, a `Map<string, number>` index should be used. This optimisation is deferred (YAGNI) — document as a comment in the implementation.

## Finding 7 — Embedding Input Text Format

**Decision**: Unchanged — `"{name}. {description}. Tags: {tags.join(', ')}"`

**Example**:
> `"Spaghetti Carbonara. A classic Roman pasta with eggs, cheese, and pancetta. Tags: italian, pasta, quick, comfort-food"`

Cuisine tags are single words and will tokenize cleanly into the vocabulary as high-signal terms.
